Agile Git

A step by step workflow proposal for real life web projects

Speaker: Antonio Pastorino
13/14 October 2018

The main picture

Real life web projects present us a set of nontrivial challenges that could be dealt through the proper use of a VCS system

As developers we need to be able to

  • keep track of our work in an efficien and effective way
  • independently deliver features in the right environment at the required pace
  • identify errors, bugs and regressions
  • postpone the delivery of a feature even when is production-ready
  • play around without breaking others code

Implementing a VCS workflow is an attempt to cope with all of these necessities respecting our clients requirements, milestones and deadlines

(and all the crazy stuff, yes)

Prerequisites

Required basic knowledge

  • Git
  • GitHub
  • Agile and at least one of its implementations (like SCRUM)

Concepts

Branch

A separate line of work

Public branch

It is a branch that more than one person pulls from

Private branch

It is a branch that you alone are using and is not exposed in the public repo

Workflow

In a nutshell, a version control workflow handles

  • features implementation
  • bug fixing
  • releases managing

Our codebase is aimed to run on multiple environments (local, dev, stage/QA, production)

From small to large to HUGE projects, keeping a clean and sane history is paramount

Git is a down-to-earth tool for designing VCS workflows rather than a mere version control system

Branching Model

Main branches

Persisting public branches that reflects the state of a specific environment

No one pushes its commits directly on these branches

... and this is quite a key concept

The minimal set of main branches is:

  • master reflects the production-ready state
  • staging reflects the qa-ready state

With the same logic we can define a persistent main branch for each environment we should need

Branching Model

Feature branches

Public branches that unlike the main ones usually have a limited lifetime

These branches should always lead to a pull request

They should be deleted and cease to exist when the corresponding pull request will eventually be merged into master

Branching Model

Task branches

These branches are likely to be private, connected with a specific feature branch and with a very limited lifetime

They should represent the local state of your development, where you can play around without worriyng about other development

From time to time you should rebase the corresponding feature branch into them and move on with your development

When you have something to deliver you should merge them into the corresponging feature branch

A little digression

Rebase

When merging hell is not enough

There are two main ways to integrate changes from one branch to another

  • merge the plain and simple way
  • rebase the "why I hate me so much" way

Merge

  • the easiest option
  • creates a new "merge commit" in the destination branch
  • this "merge commit" ties together the histories of both branches
  • this is a non-destructive operation
  • existing branches are not changed in any way
  • on the other hand the destination branch will have an extraneous "merge commit"

Rebase

  • moves the entire destination branch to begin of the tip of the origin branch
  • does not create a "merge commit" but rewrites instead the project history
  • this operation is made by creating new commits
  • this way the project history is (much) cleaner
  • does not clog your log with "merge commits"
  • results ina linear project history
  • on the other hand, re-writing project history can be potentially catastrophic
  • we lose a little bit of context provided by the merge commit

Rebase pitfalls

  • rebasing results in brand new commits replacing other commits
  • this only happens in our local repository though
  • all other developers are still working with the old set of commits
  • this easily leads to
    • diverging project history
    • extra merge commits
    • two set of commits containing the same changes
    • and catastrophe

So, before you run git rebase, always ask yourself, “Is anyone else looking at this branch?”

If the answer is yes, take your hands off the keyboard and start thinking about a non-destructive way to make your changes

Otherwise, you’re safe to re-write history as much as you like.

Branching Pattern

Implementing the workflow

Let's pick a common situation, we have two main branches

  • master contains everything we want to have in production
  • staging contains everything we have in production plus some stuff the client is evaluating

We need to implement a new feature, let's say: links should be green no matter what

Get going with a clean start 

$ git checkout master
$ git pull origin master

Make a feature branch to work in 

$ git checkout -b features_all_link_green

Work hard on your feature

Commit your changes when you have something 

$ git commit -am "Some links are now green"

And finalize your work 

$ git commit -am "Now all links are green"

When you are done, you should firstly merge with master 

$ git checkout master
$ git pull origin master
$ git checkout -b features_all_link_green
$ git rebase master

Before pushing your changes, this is the very last change to squash some unwanted commit message 

$ git rebase -i master

When everything in your local env is done, is time to deliver the feature branch into the wild 

$ git push origin features_all_link_green

and open a pull request for this feature branch

Opening a pull request should leads to a merge of the branch into staging and a consequent staging release 

$ git checkout staging
$ git pull origin staging
$ git merge features_all_link_green
$ git push origin staging

Waiting for client approval and release scheduling

When we are ready for a production release, it's time to resolve the pull request 

$ git checkout master
$ git pull origin master
$ git merge features_all_link_green
$ git push origin master
$ git tag -a 1.0.X

We can also eventually delete our branch 

$ git branch -d features_all_link_green
$ git push origin :features_all_link_green

Another little digression

Pull Requests

A structured way to introduce modifications

A pull request is a method of submitting contributions to an open development project.

It is important to note that “pull requests” are a workflow method, and are not a feature of the version control system itself.

Once a pull request has been submitted by a contributor, it is then the responsibility of the project maintainers to evaluate and, where appropriate, merge it.

GitHub provides a view of the diff describing each pull request to make the process easier.

It is even possible to accept the pull request and merge it automatically.

Branching Pattern

Wrapping up

Summing up:

  • Checkout and pull the master
  • Make a new feature branch from there
  • Work on your feature and commit your changes, frequently merging the master in your branch
  • When you are done
    • merge master in your branch
    • squash unwanted commits and rewrite messages rebasing
    • push your feature branch
  • Open a pull request
  • Wait for a staging release and client approval
  • Resolve the pull request and release in production
  • Eventually delete the feature branch
